# 🎯 Trouble Shooting
## Chromedriver 메모리 누수 문제 해결

### 문제 원인
Chromedriver에서 메모리 누수가 발생한 원인은 두 가지로 파악되었습니다. <br>
<br>
driver.quit() 호출 로직에서 발생한 코드상의 문제. <br>
Dockerfile에서 Chromedriver 버전 관리 미흡으로 인한 환경 불안정성. <br>


### 해결 과정 <br>
실행 로직 개선: driver.quit() 호출과 관련된 로직을 정밀 분석하여 불필요한 메모리 점유를 방지하도록 수정. <br>
Dockerfile 최적화: 안정된 Chromedriver 버전을 명시적으로 관리하기 위한 코드 리뷰를 진행하고, 정확한 버전을 명시하여 빌드 안정성 확보. <br>


### 결과
크롤링 작업 중 발생하던 메모리 누수 문제가 해결되어 작업 안정성이 크게 향상되었습니다. <br>
Dockerfile에서의 명시적 버전 관리를 통해 다양한 환경에서도 일관된 동작을 보장할 수 있게 되었습니다. <br>


## uWSGI Worker 메모리 누수 대응
### 문제 원인
uWSGI Worker에서 메모리 사용량이 점진적으로 증가하여 일정 임계치를 초과하면 서비스 장애로 이어질 가능성이 있었습니다. <br>
이는 장시간 실행되는 프로세스에서 메모리 누수가 누적된 결과로 확인되었습니다. <br>


### 해결 과정 <br>
reload-on-as 옵션 추가: <br>
uWSGI 설정에 reload-on-as 옵션을 추가하여 메모리 사용량이 설정된 임계치를 초과할 경우 Worker를 자동으로 재시작하도록 구성했습니다. <br>


### 결과<br>
메모리 누수가 발생하더라도 자동으로 Worker를 재시작해 문제를 효율적으로 관리할 수 있게 되었습니다. <br>
Worker 재시작으로 인한 서비스 중단 가능성도 로드 밸런서를 통한 트래픽 분산으로 문제없이 처리되었습니다. <br>


### 적용 결과 및 효과<br>
#### Chromedriver<br>
메모리 누수 문제를 근본적으로 해결하여 크롤링 작업의 안정성을 확보. <br>
다양한 환경에서 일관된 동작이 가능해짐. <Br>
<br>
#### uWSGI<br>
메모리 사용량 관리 강화로 장기 실행 환경에서도 안정적 서비스 유지. <br>
임계치 초과 시 자동 재시작을 통해 메모리 누수를 효과적으로 관리. <br>
로드 밸런서 <br>
uWSGI Worker 재시작 시에도 트래픽을 로드 밸런서를 통해 분산하여 서비스 연속성을 보장. <br>


### 결론<br>
이번 메모리 누수 해결 작업을 통해 전반적인 시스템 성능과 안정성을 크게 향상시킬 수 있었습니다. <br>
특히, Chromedriver와 uWSGI에서의 문제를 체계적으로 분석하고 해결한 경험은 다양한 운영 환경에서 발생할 수 있는 문제를 대비하는 데 중요한 교훈을 제공했습니다<br> <br>

## NTT Ncloud 환경 서버 이중화
NKS에 배포된 NTT 서버를 Ncloud 환경에서도 동작하도록 설정하여 이중화 작업을 진행했습니다.<br>
그러나 Ncloud와 NKS 환경이 완전히 동일하지 않아 기존 Dockerfile이 예상대로 동작하지 않는 문제가 발생했습니다.<br>

이 문제를 해결하기 위해 두 환경의 차이점을 분석한 후, 최대한 유사한 설정을 유지하면서도 각 환경에 맞춘 별도의 Dockerfile을 작성하였습니다. <br>
이를 통해 두 환경 모두에서 안정적으로 서버를 운영할 수 있게 되었습니다.<br>

### 주요 작업 내용
#### 이중화 목적<br>
NKS와 Ncloud 환경에 동일한 서비스를 배포하여 고가용성 확보. <br>
#### 문제점 <br>
환경 차이로 인해 기존 Dockerfile이 Ncloud에서 동작하지 않음. <br>
#### 해결 방안 <br>
- 두 환경의 차이점 분석(NKS와 Ncloud의 OS, 라이브러리, 네트워크 설정 등). <br>
- Ncloud와 NKS 환경별로 최적화된 Dockerfile을 작성. <br>
- 최대한 환경을 유사하게 유지하면서도 각 환경에 맞춘 설정 적용. <br>

결과적으로, 두 환경에서의 안정적인 서비스 동작을 보장하며 이중화 작업을 성공적으로 완료하였습니다. <br>
이러한 경험은 다양한 클라우드 환경에서의 이식성과 호환성을 고려한 DevOps 작업의 중요성을 다시금 확인하는 계기가 되었습니다. <br><br>

## NTT K8S Nat 설정 
쿠버네티스(Kubernetes) 환경에서 애플리케이션을 배포하는 경우, Pod가 항상 유동적인 IP를 가지기 때문에 ACL(Access Control List) 처리가 어려운 문제를 겪을 수 있습니다. <br>
이를 해결하기 위해, NAT(Network Address Translation)를 설정하여 컨테이너의 IP가 고정적으로 할당되도록 구성했습니다.<br>
<br>
그러나, 작업 도중 실제로 Pod가 NAT를 거쳐 올바른 Destination IP에 도달하는지 확인하기 어려운 상황에 직면했습니다.  <br>
이를 해결하기 위해 간단한 웹 서버를 개발하였고, curl 요청을 통해 Remote Address와 Host 정보를 확인하는 방식을 도입했습니다. <br>
이를 통해 NAT 설정이 올바르게 작동하고 있음을 검증할 수 있었습니다. <br>
<br>
### 주요 작업 내용 <br>
#### 문제점 <br>
Pod의 유동적인 IP로 인해 ACL 처리 및 트래픽 제어가 어려움. <br>
#### 해결책<br>
NAT를 활용하여 컨테이너에 고정 IP 할당. <br>
#### 검증 과정 <br> 
- 간단한 웹 서버를 생성. <br>
- curl 요청을 통해 Remote Address와 Host 정보를 확인. <br>
- NAT 설정이 의도한 대로 작동함을 확인.<br>
이 과정을 통해 Kubernetes 환경에서 네트워크 안정성을 확보하였으며, ACL 및 트래픽 제어가 더욱 용이해졌습니다. <br>
해당 솔루션은 Kubernetes 기반 시스템의 배포 및 관리 효율성을 크게 향상시키는 데 기여했습니다.<br><br>

## Kolo Jenkins 배포 문제 
현재 Jenkins와 Kolo 서버가 각각 Docker 컨테이너 환경에서 운영되고 있으며, Jenkins 컨테이너를 통해 Kolo 서버를 재가동하는 작업이 필요했습니다. <br>
이를 위해 Docker 컨테이너 간 연동을 구성하고, docker.sock 공유를 통해 Jenkins 컨테이너에서 Kolo 서버의 컨테이너를 제어할 수 있도록 설정하였습니다. <br>

### 문제 상황
컨테이너 간 연동 설정은 성공적으로 완료되었으나, 재가동 과정에서 일부 컨테이너(nginx, uWSGI)가 정상적으로 **stop**되지 않는 문제가 발생했습니다.<br>
또한, 재가동 시 기존 컨테이너 이름과 중복되는 오류로 인해 재배포가 실패하는 상황이 발생했습니다.<br>

### 해결 방안
컨테이너의 정상적인 재가동을 보장하기 위해 다음과 같은 방안을 도입하였습니다. <br>
- SSH 기반 재배포 방식 전환

Jenkins 컨테이너에서 직접 Kolo 서버의 컨테이너를 제어하는 대신, 서버 내부로 SSH 연결을 설정하여 필요한 컨테이너 작업을 수행하도록 변경하였습니다.
이를 통해 중복 컨테이너 이름 문제와 stop 실패로 인한 오류를 방지할 수 있었습니다.
### 컨테이너 관리 로직 개선
기존 컨테이너가 정상적으로 종료되지 않는 문제를 해결하기 위해, 재가동 전에 실행 중인 컨테이너를 강제 종료하도록 관리 스크립트를 개선하였습니다. <br>
docker stop 및 docker rm 명령을 사용하여 기존 컨테이너를 명확히 정리한 뒤, 새로운 컨테이너를 배포하도록 변경하였습니다.<br>
### 주요 개선 내용 <br>
- docker.sock 공유를 통한 초기 접근성 확보: Jenkins 컨테이너에서 Docker API 호출이 가능하도록 설정.<br>
- SSH 방식 전환: 서버 내에서 직접 컨테이너를 제어하는 안정적인 배포 프로세스 도입. <br>
- 오류 방지 로직 추가: 중복된 컨테이너 이름 문제를 해결하고, 컨테이너 종료 실패에 대비한 강제 정리 프로세스 적용. <br>
### 결과 및 효과 <br>
Jenkins와 Kolo 서버 간의 원활한 연동을 유지하며, 배포 과정의 신뢰성을 높였습니다. <br>
컨테이너 관리 및 배포 절차를 자동화하면서도, 예상치 못한 오류에 대한 유연한 대응이 가능해졌습니다. <br>
이 작업을 통해 전체 배포 프로세스의 안정성과 운영 효율성을 크게 향상시켰습니다. <br><br>

## NTT Query 최적화 
Django 애플리케이션에서 데이터를 저장할 때, for문을 통해 개별적으로 save() 메서드를 호출하면 데이터의 양(N)에 비례하여 처리 시간이 급격히 증가합니다. <br>
이로 인해 사용자에게 데이터를 반환하기까지 시간이 오래 걸리는 문제가 발생했습니다.<br>
<br>
이를 해결하기 위해 Django에서 제공하는 **bulk_create**와 bulk_update 메서드, 그리고 bulk 라이브러리를 활용하여 대량 데이터를 묶음으로 처리하는 방식으로 쿼리를 최적화했습니다. <br>

### 문제점
- save() 호출 방식:
데이터를 한 건씩 저장하는 방식은 N번의 데이터에 대해 N개의 SQL 쿼리가 실행되므로, 쿼리 오버헤드로 인해 저장 시간이 과도하게 소요됩니다.

- 사용자 경험 악화:
대량 데이터 저장 시 처리 시간이 길어져 사용자에게 데이터를 반환하는 데 지연이 발생.

### 해결 방안
**1. bulk_create 활용** <Br>
다수의 객체를 한 번의 SQL 쿼리로 저장. <br>
장점: 데이터 삽입 속도 대폭 향상.<br>
예제 코드:<br>

- 기존 방식
``` python
for data in data_list:  
    MyModel.objects.create(**data)  
```

- 개선 방식  
```
MyModel.objects.bulk_create([MyModel(**data) for data in data_list])
```

**2. bulk_update 활용**
기존 데이터의 특정 필드를 대량으로 업데이트할 때 사용.<br>

장점: 여러 개의 SQL 업데이트 문을 하나로 합쳐 실행.<br>
예제 코드:<br>

- 업데이트 대상 객체 리스트  
``` python
objects_to_update = [  
    MyModel(id=1, field="new_value"),  
    MyModel(id=2, field="another_value"),  
]  
```
- 대량 업데이트
```
MyModel.objects.bulk_update(objects_to_update, ['field'])
```

**3. bulk 라이브러리 활용**<br>
- Django 기본 기능 외에도 대량 데이터를 더 유연하게 처리할 수 있는 오픈소스 라이브러리.<br>
- 특히 데이터 삽입과 업데이트를 혼합 처리해야 하는 상황에서 유용.<br>
  
### 결과 및 효과
쿼리 최적화: N번 실행되던 쿼리가 1~2번으로 줄어들어 성능이 크게 향상. <br>
반응 속도 개선: 사용자에게 데이터를 반환하는 시간이 단축되어 UX가 향상됨. <br>
대량 데이터 처리 효율성: 대규모 데이터 작업에서도 안정적이고 빠른 처리가 가능해짐. <br>
이번 최적화를 통해 Django의 기본 기능과 추가 라이브러리를 활용하여 대량 데이터를 효율적으로 관리하는 방법을 구현했습니다. <br>
